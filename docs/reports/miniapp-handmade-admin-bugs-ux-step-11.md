# Mini App Handmade - Step 11 Report (Admin Bugs + UX Restructure)

## Какие баги исправлены

1. Исправлен `invalid input syntax for type uuid: "guest_local"`:
- Причина: при запуске спина в розыгрыше в `giveaway_results.profile_id` отправлялся `currentProfile.id`.
- В Telegram runtime активный профиль мог быть `guest_local`, а поле в БД ожидает `uuid`.
- Исправление:
  - в `AppContext` перед вставкой результата спина теперь передается `profileId` только если это валидный UUID;
  - в `supabase-repository` добавлена дополнительная серверная защита: `profile_id` приводится к `null`, если вход не UUID.

2. Исправлен `Cannot coerce the result to a single JSON object`:
- Причина: репозиторий использовал хрупкий `.single()` в admin-mutation сценариях, где фактически могло прийти 0 строк (например, нет доступа по policy или запись не найдена).
- Исправление:
  - критичные mutation-запросы переведены на `maybeSingle()`;
  - добавлен `unwrapSingleRow(...)` с явной обработкой 0-row/доступа и нормальными сообщениями вместо падения.

3. Исправлены admin-действия по товарам:
- проверен и исправлен поток флагов товара (видимость / под заказ / рекомендация) через устойчивый mutation-path;
- изменения проходят через backend persistence и не теряются в state после операции.

4. Исправлена логика CTA «Приобрести»:
- на карточке товара и странице товара для обычного пользователя CTA работает корректно;
- для admin/user-seller CTA покупки скрыт как бессмысленный сценарий.

5. Перестроено управление розыгрышем:
- управление сессиями и лотами сосредоточено на странице `Giveaway`;
- исправлены действия по статусам сессии (`Запустить сессию`, `Завершить сессию`);
- исправлено удаление лотов;
- исправлен контролируемый ввод длительности спина.

## Как изменена логика admin profile

- Профиль разделен на 2 режима:
  - user profile (избранное, о мастере, связь с продавцом);
  - admin profile (чистый admin-hub).
- В admin-профиле добавлена структурированная навигация и логические блоки:
  - витрина/каталог/розыгрыш;
  - admin panel по вкладкам;
  - компактный служебный status-блок.
- Информационный хаос убран, длинный смешанный список заменен на разделенные блоки.

## Какие admin-действия перенесены на страницу розыгрыша

- Добавление лотов в сессию.
- Удаление лотов из сессии.
- Управление статусами сессии и запуск спина.
- В админ-панели оставлена ссылка на страницу розыгрыша, но не управление составом лотов.

## Упрощение admin settings

- В блоке настроек бренда оставлены только:
  - название бренда,
  - слоган,
  - Hero badge.
- Лишние поля убраны из этого раздела.

## Ограничения

- Полная проверка end-to-end внутри реального Telegram WebView требует runtime-проверки после деплоя.
- Если в целевом Supabase окружении RLS/профили админа настроены неполно, часть admin-mutation может быть отклонена policy-уровнем.
- Минимальный внешний доступ для полного закрытия риска:
  - runtime доступ к deployed Mini App в Telegram;
  - подтвержденная запись admin-профиля в `profiles`/policy-совместимом auth-контексте.
