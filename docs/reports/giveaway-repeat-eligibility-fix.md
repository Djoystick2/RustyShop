# Giveaway Repeat Eligibility Fix

Date: 2026-03-13

## Причина

После разбора текущей реализации выяснилось следующее:

- Список товаров, доступных для добавления в giveaway-сессию, уже строился по `giveaway_items` текущей выбранной сессии, а не по полной истории `giveaway_results`.
- Защита от повторного выпадения уже была локальной для одной сессии: после спина `createGiveawayResult` переводит только текущий `giveaway_item.isActive` в `false`.
- Проблема была в разрозненности логики и в admin UI: правило eligibility было продублировано в нескольких местах, а продуктовая админка показывала history-based блокировку как жёсткое состояние на поверхности.

## Что изменено

1. Вынесено общее правило eligibility в domain helper:
   - `src/domain/giveaway/wheel.ts`
   - добавлена функция `listProductsAvailableForGiveawaySession(products, sessionItems)`
   - она явно исключает только товары, уже прикреплённые к текущей сессии
   - история прошлых побед сознательно не участвует в фильтрации

2. На общий helper переведены обе giveaway-поверхности:
   - `src/pages/GiveawayPage.tsx`
   - `src/components/admin/AdminPanel.tsx`

3. В admin UI убрана жёсткая history-based блокировка кнопки удаления товара:
   - кнопка больше не выглядит как глобальный stop-state из-за прошлой победы
   - при попытке удалить товар из истории админ получает явное сообщение, что удаление ограничено ради сохранения истории, но это не мешает повторно использовать товар в новой giveaway-сессии

## Что сохранено намеренно

- Модель `draft / active / completed` не менялась.
- Повторное выпадение одного и того же лота внутри одной и той же сессии по-прежнему блокируется через `giveaway_item.isActive`.
- История `giveaway_results` не удаляется и не переписывается.
- Ручное удаление лота из сессии и остальной giveaway flow не менялись.

## Изменённые файлы

- `src/domain/giveaway/wheel.ts`
- `src/pages/GiveawayPage.tsx`
- `src/components/admin/AdminPanel.tsx`

## Проверки

- `npm run typecheck`
- `npm run build`

## Ограничения

- Ограничение на удаление товара из общей product-базы, если он уже участвует в исторических giveaway results, сохранено намеренно. Это отдельное ограничение целостности истории, а не правило eligibility для новых giveaway-сессий.
- В Supabase-схеме `giveaway_results.product_id` остаётся с `on delete restrict`, поэтому полное снятие delete-ограничения потребовало бы отдельной backend-миграции и пересмотра того, как хранить исторические связи товара с результатами.
